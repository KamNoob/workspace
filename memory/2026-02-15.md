## Fixed: OpenClaw TUI Authentication Issue

### Problem
TUI was hanging/not responding when launched. Investigation revealed authentication was blocked.

### Root Cause
**Expired Claude Pro OAuth token** - not just cooldown state. The token in auth-profiles.json (`sk-ant-oat01-...`) had expired, causing HTTP 401 errors from Anthropic API.

**Key insight**: Cooldown was a symptom, not the cause. OpenClaw correctly applied cooldown after repeated auth failures. Clearing cooldown alone didn't fix the issue - needed a fresh, valid token.

### Solution Steps
1. **Got fresh Claude Pro OAuth token**
   - Used Claude Code CLI: `ccode auth login` → `ccode auth token`
   - Added via: `openclaw models auth paste-token`
   - Created new profile: `anthropic:manual`

2. **Updated auth-profiles.json**
   - Changed `lastGood.anthropic` from `anthropic:default` to `anthropic:manual`
   - Cleared cooldown state from old profiles (set usageStats to `{}`)
   - Added empty usageStats entry for new profile

3. **Restarted gateway**
   - `openclaw gateway restart` to reload auth configuration

4. **Verified**
   - Tested with: `openclaw agent --agent main --message "test"`
   - Confirmed successful authentication (errorCount: 0, lastUsed timestamp set)

### Key Learnings

**Authentication Troubleshooting:**
- When TUI/agent hangs, check `auth-profiles.json` for cooldown state AND actual token validity
- HTTP 401 errors = invalid/expired token (not network/API issues)
- Cooldown mechanism protects API from being hammered with invalid credentials
- Always verify root cause before clearing cooldown

**Claude Pro Token Management:**
- OAuth tokens can be obtained from **any machine** with Claude Code CLI, not just the OpenClaw server
- Token is account-based, not machine-specific
- Recommended workflow: maintain Claude Code CLI on main dev machine, copy tokens as needed
- Token lifespan: several months to a year before refresh required
- Transfer securely (SSH, encrypted channels)

**OpenClaw Auth Profile Structure:**
```json
{
  "profiles": { "provider:name": { "type": "token", "token": "..." } },
  "lastGood": { "provider": "provider:profile-name" },  // Points to which profile to use
  "usageStats": { "provider:name": { "lastUsed": timestamp, "errorCount": 0 } }
}
```

**Post-token-update checklist:**
1. Verify new profile exists in `profiles` section
2. Update `lastGood.{provider}` to point to new profile name
3. Clear cooldown state (remove errorCount, failureCounts, cooldownUntil fields)
4. Restart gateway
5. Test authentication with simple agent command

### memory-lancedb Plugin Limitations Discovered

**Investigation:** Attempted to switch memory-lancedb to local Ollama embeddings (nomic-embed-text)

**Results:**
- Plugin is **hardcoded** to only accept OpenAI models
- Validation in `config.ts` enforces specific model names:
  - `text-embedding-3-small` (1536 dims)
  - `text-embedding-3-large` (3072 dims)
- Cannot use local models without modifying plugin source code

**Attempted workaround:**
- Set `OPENAI_BASE_URL=http://127.0.0.1:11434/v1` to route to Ollama
- Set model to `nomic-embed-text`
- **Failed:** Config validation rejects unknown model names
- Gateway failed to start with config error

**Dimensional mismatch:**
- OpenAI text-embedding-3-small: 1536 dimensions
- Ollama nomic-embed-text: 768 dimensions
- Even with API routing, dimension mismatch would cause issues
